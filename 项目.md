# 门店巡检平台 — 项目分析与核心知识点

## 一、项目概述

门店巡检平台是一个 **多租户 SaaS 系统**，用于连锁企业管理门店的定期巡检工作。核心流程：总部创建巡检任务 → 巡检员到店执行 → 门店负责人提交整改 → 巡检员/管理员复核关闭。

### 核心角色

| 角色 | 标识 | 权限 |
|------|------|------|
| 总部管理员 | `hq_admin` | 创建任务、查看所有数据、审计日志、审核通过/打回 |
| 巡检员 | `inspector` | 创建任务、提交巡检结果、审核通过/打回 |
| 门店负责人 | `store_manager` | 查看本店任务、提交整改（含附件上传） |

## 四、数据模型（Prisma Schema）

### 4.1 核心模型关系

```
Company (企业)
  ├── Store[] (门店)
  ├── User[] (用户)
  ├── InspectionTask[] (巡检任务)
  ├── Attachment[] (附件)
  └── AuditLog[] (审计日志)

User
  ├── role: UserRole  (hq_admin | inspector | store_manager)
  └── storeId?: Int   (store_manager 绑定具体门店)

InspectionTask
  ├── status: TaskStatus
  ├── InspectionItem[] (检查项)
  └── RectificationSubmission[] (整改记录)

RectificationSubmission
  └── Attachment[] (整改附件)
```

### 4.2 多租户隔离

每张业务表都有 `companyId` 字段。所有查询 **必须** 带上 `companyId` 条件：

```typescript
// 正确：带 companyId 隔离
prisma.inspectionTask.findFirst({
  where: { id: taskId, companyId: auth.companyId }
})

// 错误：遗漏 companyId，会读到其他企业数据
prisma.inspectionTask.findFirst({ where: { id: taskId } })
```

`companyId` 来自 JWT payload，不依赖前端传参，防止越权。

## 五、认证机制（JWT）

### 5.1 登录流程

```
POST /api/auth/login
  1. Zod 校验 email + password
  2. prisma.user.findUnique({ where: { email } })
  3. bcrypt.compare(password, user.passwordHash)
  4. signToken({ userId, companyId, role, storeId })  → JWT
  5. 返回 { token, user }
```

### 5.2 JWT Payload 结构

```typescript
interface JWTPayload {
  userId: number
  companyId: number
  role: 'hq_admin' | 'inspector' | 'store_manager'
  storeId?: number   // 仅 store_manager 有值
}
```

### 5.3 请求鉴权

```typescript
// middleware/auth.ts
export async function requireAuth(req: NextRequest): Promise<JWTPayload | NextResponse> {
  const authHeader = req.headers.get('authorization')
  if (!authHeader?.startsWith('Bearer ')) {
    return NextResponse.json({ error: '未授权' }, { status: 401 })
  }
  const token = authHeader.slice(7)
  const payload = verifyToken(token)   // 验证签名 + 过期
  if (!payload) {
    return NextResponse.json({ error: 'Token 无效或已过期' }, { status: 401 })
  }
  return payload
}
```

**关键点**：`requireAuth` 返回联合类型 `JWTPayload | NextResponse`，调用方通过 `instanceof NextResponse` 判断是否认证失败：

```typescript
const auth = await requireAuth(req)
if (auth instanceof NextResponse) return auth  // 直接返回 401 响应
// auth 此时已被 TypeScript 收窄为 JWTPayload
const { userId, companyId, role } = auth
```

### 5.4 密码安全

```typescript
// 注册时
const hash = await bcrypt.hash(password, 10)  // 10 轮 salt

// 登录时
const ok = await bcrypt.compare(plaintext, hash)
```

bcrypt 是**单向哈希**，无法反推原始密码。`hash` 自带 salt，同一密码每次哈希结果不同。

## 七、任务状态机

```
PENDING_INSPECTION（待巡检）
        │ 巡检员/管理员：提交巡检结果（有问题）
        ▼
PENDING_RECTIFICATION（待整改）
        │ 门店负责人：提交整改报告 + 附件
        ▼
PENDING_REVIEW（待复核）
        │ 巡检员/管理员：审核
        ├──[通过]──▶ CLOSED（已关闭）
        └──[打回]──▶ PENDING_RECTIFICATION（重新整改）
```

**状态机的价值**：
- 服务端强制执行，前端无法绕过
- 非法流转返回 422 状态码
- 每次流转写入审计日志（before/after JSON）

## 八、AWS S3 预签名 URL 文件上传

### 8.1 为什么用预签名 URL？

直接上传需要文件经过服务器中转，浪费带宽且增加服务器压力。预签名 URL 让客户端**直接与 S3 通信**：

```
Browser ──[1. 请求上传凭证]──▶ Server
Server  ──[2. 返回预签名 URL + s3Key]──▶ Browser
Browser ──[3. PUT 文件直传 S3]──▶ AWS S3
Browser ──[4. 提交 s3Key 到 Server]──▶ Server
Server  ──[5. 保存 Attachment 记录]──▶ DB
```

### 8.2 上传流程代码

```typescript
// 第一步：获取预签名 URL
const presignRes = await fetch('/api/files/presign-upload', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json', Authorization: `Bearer ${token}` },
  body: JSON.stringify({ taskId, fileName, mimeType, size }),
})
const { uploadUrl, s3Key } = await presignRes.json()

// 第二步：直传 S3（不经过我们的服务器）
await fetch(uploadUrl, {
  method: 'PUT',
  body: file,
  headers: { 'Content-Type': file.type },
})

// 第三步：提交整改，附带 s3Key
await fetch(`/api/tasks/${taskId}/rectifications`, {
  method: 'POST',
  body: JSON.stringify({ note, s3Keys: [s3Key] }),
})
```

### 8.3 服务端生成预签名 URL

```typescript
import { S3Client, PutObjectCommand } from '@aws-sdk/client-s3'
import { getSignedUrl } from '@aws-sdk/s3-request-presigner'

const s3 = new S3Client({ region: process.env.AWS_REGION })

export async function getPresignedUploadUrl(key: string, contentType: string, expiresIn = 300) {
  const command = new PutObjectCommand({
    Bucket: process.env.AWS_S3_BUCKET,
    Key: key,
    ContentType: contentType,
  })
  return getSignedUrl(s3, command, { expiresIn })  // URL 5分钟有效
}
```

### 8.4 S3 Key 命名规范

```
company/{companyId}/task/{taskId}/{timestamp}-{random}.{ext}
```

按 companyId 分目录，天然实现文件隔离，便于 IAM 策略精细控制。

## 九、Prisma ORM 核心用法

### 9.1 事务（$transaction）

整改提交需要原子性：创建记录 + 更新状态 + 写审计日志，要么全成功要么全回滚：

```typescript
const result = await prisma.$transaction(async (tx) => {
  const sub = await tx.rectificationSubmission.create({ data: {...} })
  await tx.attachment.createMany({ data: [...] })
  await tx.inspectionTask.update({ where: { id: taskId }, data: { status: 'PENDING_REVIEW' } })
  await tx.auditLog.create({ data: {...} })
  return sub
})
```

### 9.2 关联查询（include）

```typescript
prisma.rectificationSubmission.findMany({
  where: { taskId },
  include: {
    attachments: true,
    submitter: { select: { id: true, email: true, role: true } },
  },
  orderBy: { submittedAt: 'desc' },
})
```

`include` 做 JOIN 查询，`select` 只返回需要的字段，避免暴露密码哈希等敏感数据。

### 9.3 聚合计数（_count）

```typescript
prisma.inspectionTask.findMany({
  include: {
    _count: {
      select: { inspectionItems: true, rectificationSubmissions: true }
    }
  }
})
// 返回 task._count.inspectionItems（不加载全部子记录，高效）
```

### 9.4 并行查询

看板指标 5 个计数并行执行，性能优于串行：

```typescript
const [total, closed, overdue, pendingRect, pendingReview] = await Promise.all([
  prisma.inspectionTask.count({ where: { companyId } }),
  prisma.inspectionTask.count({ where: { companyId, status: 'CLOSED' } }),
  prisma.inspectionTask.count({ where: { companyId, dueDate: { lt: now }, status: { not: 'CLOSED' } } }),
  prisma.inspectionTask.count({ where: { companyId, status: 'PENDING_RECTIFICATION' } }),
  prisma.inspectionTask.count({ where: { companyId, status: 'PENDING_REVIEW' } }),
])
```

### 10.4 'use client' 与服务端组件

- 文件顶部写 `'use client'` → 客户端组件，可用 `useState`/`useEffect`，但不能直接访问服务器资源
- 无指令默认为服务端组件，在服务器端渲染，不能用浏览器 API
- API Routes（`route.ts`）始终在服务器端执行，无需指令

## 十一、前端实现要点

### 11.1 单页应用结构（无路由库）

```typescript
type Tab = 'dashboard' | 'tasks' | 'audit'
const [tab, setTab] = useState<Tab>('dashboard')

// 条件渲染实现 Tab 切换
{tab === 'dashboard' && <DashboardTab ... />}
{tab === 'tasks'     && <TasksTab ... />}
{tab === 'audit'     && <AuditTab ... />}
```

### 11.2 useCallback 避免重渲染

`loadTasks` 被 `useEffect` 依赖，若不用 `useCallback` 包裹，每次渲染都会创建新函数引用，导致 `useEffect` 无限触发：

```typescript
const loadTasks = useCallback(async (authToken = token) => {
  // ...fetch logic
}, [token, filterStatus, filterOverdue])  // 依赖项变化时才重建函数
```

### 11.3 useMemo 缓存计算值

```typescript
const isAuthed = useMemo(() => Boolean(token && user), [token, user])
```

### 11.4 联合类型 + 类型收窄

```typescript
// 返回 JWTPayload | NextResponse 的函数
const auth = await requireAuth(req)
if (auth instanceof NextResponse) return auth
// TypeScript 自动收窄：auth 此后为 JWTPayload 类型
const { role } = auth  // 无需类型断言
```

## 十二、Zod 输入校验

```typescript
const createTaskSchema = z.object({
  storeId: z.number().int().positive(),
  title: z.string().min(1).max(100),
  description: z.string().optional(),
  assigneeId: z.number().int().positive(),
  dueDate: z.string().datetime(),
  items: z.array(z.object({ itemName: z.string().min(1) })).min(1),

const parsed = createTaskSchema.safeParse(body)
if (!parsed.success) {
  return NextResponse.json({ error: parsed.error.flatten() }, { status: 400 })

// parsed.data 已通过类型推导，不需要手动标注类型
const { storeId, title, items } = parsed.data
```

## 十三、审计日志设计

每次重要操作（状态变更、任务创建）在事务内写入 `AuditLog`：

```typescript
await tx.auditLog.create({
  data: {
    companyId,
    entityType: 'inspection_task',   // 操作的实体类型
    entityId: taskId,                 // 实体 ID
    action: 'STATUS_CHANGE',          // 操作类型
    beforeJson: { status: 'PENDING_INSPECTION' },
    afterJson:  { status: 'PENDING_RECTIFICATION' },
    operatorId: userId,               // 操作人
  },
})
```

与业务操作在**同一事务**中：业务失败则审计日志也不写入，保证一致性。

## 十五、关键设计模式总结

| 模式 | 实现 | 目的 |
|------|------|------|
| 多租户隔离 | 每查询带 `companyId` | 防止数据越权访问 |
| 状态机 | 服务端 `VALID_TRANSITIONS` | 保证业务流程完整性 |
| 预签名 URL | S3 SDK `getSignedUrl` | 文件直传，减轻服务器压力 |
| 事务原子性 | Prisma `$transaction` | 业务+审计要么全成功 |
| JWT 无状态认证 | `jsonwebtoken` | 服务端无需存储 Session |
| 类型安全 | TypeScript + Zod | 编译时 + 运行时双重保障 |
| 并行查询 | `Promise.all` | 减少看板接口响应时间 |
| 权限收窄 | `instanceof` 类型守卫 | TypeScript 自动类型推导 |
