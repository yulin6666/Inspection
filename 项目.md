# Store Inspection Platform — Project Analysis & Key Concepts

## 1. Project Overview

The Store Inspection Platform is a **multi-tenant SaaS system** for chain enterprises to manage periodic store inspections. Core flow: HQ creates inspection task → inspector visits store → store manager submits rectification → inspector/admin reviews and closes.

### Core Roles

| Role | Identifier | Permissions |
|------|-----------|-------------|
| HQ Admin | `hq_admin` | Create tasks, view all data, audit logs, approve/reject |
| Inspector | `inspector` | Create tasks, submit inspection results, approve/reject |
| Store Manager | `store_manager` | View own store tasks, submit rectifications (with file upload) |

## 4. Data Model (Prisma Schema)

### 4.1 Core Model Relationships

```
Company
  ├── Store[]
  ├── User[]
  ├── InspectionTask[]
  ├── Attachment[]
  └── AuditLog[]

User
  ├── role: UserRole  (hq_admin | inspector | store_manager)
  └── storeId?: Int   (store_manager is bound to a specific store)

InspectionTask
  ├── status: TaskStatus
  ├── InspectionItem[]
  └── RectificationSubmission[]

RectificationSubmission
  └── Attachment[]
```

### 4.2 Multi-Tenant Isolation

Every business table has a `companyId` column. All queries **must** include a `companyId` condition:

```typescript
// Correct: scoped by companyId
prisma.inspectionTask.findFirst({
  where: { id: taskId, companyId: auth.companyId }
})

// Wrong: missing companyId — would expose data from other companies
prisma.inspectionTask.findFirst({ where: { id: taskId } })
```

`companyId` comes from the JWT payload, not from frontend parameters, preventing privilege escalation.

## 5. Authentication (JWT)

### 5.1 Login Flow

```
POST /api/auth/login
  1. Zod validate email + password
  2. prisma.user.findUnique({ where: { email } })
  3. bcrypt.compare(password, user.passwordHash)
  4. signToken({ userId, companyId, role, storeId })  → JWT
  5. return { token, user }
```

### 5.2 JWT Payload Structure

```typescript
interface JWTPayload {
  userId: number
  companyId: number
  role: 'hq_admin' | 'inspector' | 'store_manager'
  storeId?: number   // only set for store_manager
}
```

### 5.3 Request Authorization

```typescript
// middleware/auth.ts
export async function requireAuth(req: NextRequest): Promise<JWTPayload | NextResponse> {
  const authHeader = req.headers.get('authorization')
  if (!authHeader?.startsWith('Bearer ')) {
    return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
  }
  const token = authHeader.slice(7)
  const payload = verifyToken(token)   // verify signature + expiry
  if (!payload) {
    return NextResponse.json({ error: 'Token invalid or expired' }, { status: 401 })
  }
  return payload
}
```

**Key point**: `requireAuth` returns a union type `JWTPayload | NextResponse`. Callers use `instanceof NextResponse` to check for auth failure:

```typescript
const auth = await requireAuth(req)
if (auth instanceof NextResponse) return auth  // return 401 response directly
// TypeScript narrows auth to JWTPayload here
const { userId, companyId, role } = auth
```

### 5.4 Password Security

```typescript
// on registration
const hash = await bcrypt.hash(password, 10)  // 10 salt rounds

// on login
const ok = await bcrypt.compare(plaintext, hash)
```

bcrypt is a **one-way hash** — the original password cannot be recovered. The hash includes a salt, so the same password produces a different hash each time.

## 6. RBAC Access Control

### 6.1 State Machine with Role Binding

```typescript
// valid status transitions
const VALID_TRANSITIONS = {
  PENDING_INSPECTION:    ['PENDING_RECTIFICATION'],
  PENDING_RECTIFICATION: ['PENDING_REVIEW'],
  PENDING_REVIEW:        ['CLOSED', 'PENDING_RECTIFICATION'],
  CLOSED:                [],
}

// roles allowed for each target status
const STATUS_ROLES = {
  PENDING_RECTIFICATION: ['hq_admin', 'inspector'],  // submit inspection
  PENDING_REVIEW:        ['store_manager'],           // submit rectification
  CLOSED:                ['hq_admin', 'inspector'],   // approve
}
```

Two-layer validation: ① is the status transition path valid; ② does the current role have permission to perform it.

### 6.2 Data Scope Isolation

`store_manager` is automatically restricted to their own store's tasks:

```typescript
if (role === 'store_manager' && storeId) {
  where.storeId = storeId  // read from JWT, not trusted from frontend params
}
```

## 7. Task State Machine

```
PENDING_INSPECTION
        │ Inspector/admin: submit inspection result (issues found)
        ▼
PENDING_RECTIFICATION
        │ Store manager: submit rectification report + attachments
        ▼
PENDING_REVIEW
        │ Inspector/admin: review
        ├──[approve]──▶ CLOSED
        └──[reject]───▶ PENDING_RECTIFICATION (re-rectify)
```

**Value of the state machine**:
- Enforced server-side — frontend cannot bypass it
- Invalid transitions return HTTP 422
- Every transition writes an audit log (before/after JSON)

## 8. AWS S3 Presigned URL File Upload

### 8.1 Why Presigned URLs?

Direct upload would route files through the server, wasting bandwidth and adding load. Presigned URLs let the client **communicate directly with S3**:

```
Browser ──[1. request upload credentials]──▶ Server
Server  ──[2. return presigned URL + s3Key]──▶ Browser
Browser ──[3. PUT file directly to S3]──▶ AWS S3
Browser ──[4. submit s3Key to Server]──▶ Server
Server  ──[5. save Attachment record]──▶ DB
```

### 8.2 Upload Flow Code

```typescript
// Step 1: get presigned URL
const presignRes = await fetch('/api/files/presign-upload', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json', Authorization: `Bearer ${token}` },
  body: JSON.stringify({ taskId, fileName, mimeType, size }),
})
const { uploadUrl, s3Key } = await presignRes.json()

// Step 2: upload directly to S3 (bypasses our server)
await fetch(uploadUrl, {
  method: 'PUT',
  body: file,
  headers: { 'Content-Type': file.type },
})

// Step 3: submit rectification with s3Key
await fetch(`/api/tasks/${taskId}/rectifications`, {
  method: 'POST',
  body: JSON.stringify({ note, s3Keys: [s3Key] }),
})
```

### 8.3 Server-Side Presigned URL Generation

```typescript
import { S3Client, PutObjectCommand } from '@aws-sdk/client-s3'
import { getSignedUrl } from '@aws-sdk/s3-request-presigner'

const s3 = new S3Client({ region: process.env.AWS_REGION })

export async function getPresignedUploadUrl(key: string, contentType: string, expiresIn = 300) {
  const command = new PutObjectCommand({
    Bucket: process.env.AWS_S3_BUCKET,
    Key: key,
    ContentType: contentType,
  })
  return getSignedUrl(s3, command, { expiresIn })  // URL valid for 5 minutes
}
```

### 8.4 S3 Key Naming Convention

```
company/{companyId}/task/{taskId}/{timestamp}-{random}.{ext}
```

Files are organized by `companyId`, providing natural file isolation and enabling fine-grained IAM policy control.

## 9. Prisma ORM Core Usage

### 9.1 Transactions ($transaction)

Rectification submission requires atomicity: create record + update status + write audit log — all succeed or all roll back:

```typescript
const result = await prisma.$transaction(async (tx) => {
  const sub = await tx.rectificationSubmission.create({ data: {...} })
  await tx.attachment.createMany({ data: [...] })
  await tx.inspectionTask.update({ where: { id: taskId }, data: { status: 'PENDING_REVIEW' } })
  await tx.auditLog.create({ data: {...} })
  return sub
})
```

### 9.2 Relational Queries (include)

```typescript
prisma.rectificationSubmission.findMany({
  where: { taskId },
  include: {
    attachments: true,
    submitter: { select: { id: true, email: true, role: true } },
  },
  orderBy: { submittedAt: 'desc' },
})
```

`include` performs a JOIN query; `select` returns only the needed fields, avoiding exposure of sensitive data like password hashes.

### 9.3 Aggregate Counts (_count)

```typescript
prisma.inspectionTask.findMany({
  include: {
    _count: {
      select: { inspectionItems: true, rectificationSubmissions: true }
    }
  }
})
// returns task._count.inspectionItems (no child records loaded — efficient)
```

### 9.4 Parallel Queries

Dashboard metrics run 5 counts in parallel, much faster than sequential:

```typescript
const [total, closed, overdue, pendingRect, pendingReview] = await Promise.all([
  prisma.inspectionTask.count({ where: { companyId } }),
  prisma.inspectionTask.count({ where: { companyId, status: 'CLOSED' } }),
  prisma.inspectionTask.count({ where: { companyId, dueDate: { lt: now }, status: { not: 'CLOSED' } } }),
  prisma.inspectionTask.count({ where: { companyId, status: 'PENDING_RECTIFICATION' } }),
  prisma.inspectionTask.count({ where: { companyId, status: 'PENDING_REVIEW' } }),
])
```

## 10. Next.js App Router API Routes

### 10.1 Route Handler Basic Structure


### 10.2 Dynamic Route Parameters (Next.js 15 Change)

In Next.js 15, `params` is a **Promise** and must be `await`ed:

```typescript
// Next.js 15 correct usage
export async function GET(
  req: NextRequest,
  { params }: { params: Promise<{ id: string }> }
) {
  const { id } = await params  // must await
}
```

### 10.3 Reading Query Parameters

```typescript
const { searchParams } = new URL(req.url)
const status = searchParams.get('status')
const page = parseInt(searchParams.get('page') || '1')
```

### 10.4 'use client' vs Server Components

- `'use client'` at the top of a file → client component, can use `useState`/`useEffect`, but cannot access server resources directly
- No directive → server component, rendered on the server, cannot use browser APIs
- API Routes (`route.ts`) always run on the server, no directive needed

### 11.1 Single-Page App Structure (No Router Library)

```typescript
type Tab = 'dashboard' | 'tasks' | 'audit'
```

### 11.2 useCallback to Prevent Re-renders

`loadTasks` is a `useEffect` dependency. Without `useCallback`, a new function reference is created on every render, causing `useEffect` to fire infinitely:

```typescript
const loadTasks = useCallback(async (authToken = token) => {
  // ...fetch logic
}, [token, filterStatus, filterOverdue])  // function only recreated when deps change
```

### 11.3 useMemo for Cached Computed Values

```typescript
const isAuthed = useMemo(() => Boolean(token && user), [token, user])
```

### 11.4 Union Types + Type Narrowing

```typescript
// function returning JWTPayload | NextResponse
const auth = await requireAuth(req)
if (auth instanceof NextResponse) return auth
// TypeScript automatically narrows: auth is JWTPayload from here
const { role } = auth  // no type assertion needed
```

## 12. Zod Input Validation

```typescript
const createTaskSchema = z.object({
  storeId: z.number().int().positive(),
  title: z.string().min(1).max(100),
  description: z.string().optional(),
  assigneeId: z.number().int().positive(),
  dueDate: z.string().datetime(),
  items: z.array(z.object({ itemName: z.string().min(1) })).min(1),
})

const parsed = createTaskSchema.safeParse(body)
if (!parsed.success) {
  return NextResponse.json({ error: parsed.error.flatten() }, { status: 400 })
}
// parsed.data is fully typed — no manual type annotation needed
const { storeId, title, items } = parsed.data
```

**Benefits**: validation logic doubles as documentation, error messages are structured, TypeScript types are inferred automatically.

## 13. Audit Log Design

Every significant operation (status change, task creation) writes to `AuditLog` within the same transaction:

```typescript
await tx.auditLog.create({
  data: {
    companyId,
    entityType: 'inspection_task',   // type of entity operated on
    entityId: taskId,                 // entity ID
    action: 'STATUS_CHANGE',          // operation type
    beforeJson: { status: 'PENDING_INSPECTION' },
    afterJson:  { status: 'PENDING_RECTIFICATION' },
    operatorId: userId,               // who performed the action
  },
})
```

Written in the **same transaction** as the business operation: if the business operation fails, the audit log is also not written, ensuring consistency.

## 15. Key Design Patterns Summary

| Pattern | Implementation | Purpose |
|---------|---------------|---------|
| Multi-tenant isolation | Every query includes `companyId` | Prevent cross-tenant data access |
| State machine | Server-side `VALID_TRANSITIONS` | Enforce business process integrity |
| Presigned URLs | S3 SDK `getSignedUrl` | Direct file upload, reduces server load |
| Transaction atomicity | Prisma `$transaction` | Business + audit either both succeed or both fail |
| JWT stateless auth | `jsonwebtoken` | No server-side session storage needed |
| Type safety | TypeScript + Zod | Compile-time + runtime double validation |
| Parallel queries | `Promise.all` | Reduce dashboard API response time |
| Permission narrowing | `instanceof` type guard | TypeScript automatic type inference |
